# Идеи развития publication workflow

Этот документ собирает идеи по развитию интерфейса и логики публикационных задач в `Ba_post_planner`, чтобы не терять рабочие мысли по мере итераций.

## Текущее состояние

Сейчас в `PublicationTasks` уже есть:

- встроенное редактирование текста публикации
- сохранение текста прямо в `contentItem`
- предпросмотр Markdown / HTML рядом с редактором
- история последних сохранённых версий текста
- handoff workflow
- critic check
- подтверждение публикации
- ручное и автоматическое сохранение метрик

## Ближайшие улучшения

### 1. Восстановление версии из истории

Что добавить:

- кнопку `Восстановить эту версию`
- возможность вернуть предыдущую редакцию текста в editor
- подтверждение перед заменой текущего текста

Зачем:

- быстро откатывать неудачные правки
- использовать историю не только как лог, но и как рабочий инструмент

### 2. Diff между версиями

Что добавить:

- визуальное сравнение `Было / Стало`
- подсветку добавленных и удалённых фрагментов
- компактный inline diff для быстрых правок

Зачем:

- проще понимать, что именно поменялось
- полезно для ручного ревью перед публикацией

### 3. Autosave с debounce

Что добавить:

- автосохранение текста после паузы в наборе
- индикатор `Сохранено` / `Сохраняем...`
- fallback-кнопку ручного сохранения

Зачем:

- уменьшить риск потери правок
- ускорить редакторский поток

### 4. Явный source-of-truth для publish-ready текста

Что уточнить и закрепить:

- где именно хранится финальный publish-ready body
- как синхронизируются:
  - `draft_text`
  - `quality_report.handoff_bundle.publication.body`
  - `workspace_context.source_content`

Зачем:

- избежать расхождений между редактором, handoff и публикацией
- уменьшить шанс “редактировали одно, публиковали другое”

### 5. Ручной reset к source content

Что добавить:

- кнопку `Вернуть из исходного контента`
- отдельный режим:
  - reset к последнему handoff body
  - reset к source file content

Зачем:

- быстро собирать новую редакцию от оригинала
- не копировать исходник руками

### 6. Статус dirty / unsaved changes на уровне задачи

Что добавить:

- визуальный индикатор несохранённых правок
- предупреждение при уходе со страницы
- предупреждение при переключении между задачами

Зачем:

- защитить пользователя от случайной потери текста

### 7. Подготовка публикации по каналам

Что добавить:

- channel-specific preview:
  - Reddit
  - LinkedIn
  - Telegram
  - Tilda
- channel constraints hints:
  - длина текста
  - допустимые ссылки
  - markdown/html restrictions

Зачем:

- видеть не абстрактный текст, а почти готовую публикацию в контексте платформы

### 8. Валидация перед сохранением / публикацией

Что добавить:

- длина текста
- пустые блоки
- битые ссылки
- запрещённые формулировки из dictionary
- missing CTA / missing asset warnings

Зачем:

- ловить ошибки до ручной публикации

### 9. История редакторских действий

Что добавить:

- кто редактировал
- когда редактировал
- какое было изменение
- отдельный timeline внутри task

Зачем:

- удобно для multi-user режима
- полезно для аудита и совместной работы

### 10. Черновики комментариев и reply workflow

Что добавить:

- draft ответа на комментарий
- quick actions:
  - сохранить как reply draft
  - отправить в MCP handoff
  - превратить в новую content idea

Зачем:

- замкнуть feedback loop прямо внутри publication workspace

## Улучшения для parser + publication связки

### 11. Прямое превращение parser insight в publication draft

Что добавить:

- action `Create draft from insight`
- перенос:
  - quote
  - pain point
  - audience wording
  - source links

Зачем:

- сократить путь от research до контента

### 12. Встраивание evidence рядом с текстом

Что добавить:

- sidebar с evidence snippets
- quick insert quotes
- quick insert objections / questions

Зачем:

- редактор пишет текст, не переключаясь между parser и publication tools

## UX-улучшения

### 13. Split view presets

Что добавить:

- режимы:
  - `Edit + Preview`
  - `Edit + Source`
  - `Preview + History`

Зачем:

- сделать экран удобнее под разные сценарии

### 14. Sticky action bar

Что добавить:

- закреплённую панель:
  - save
  - critic check
  - prepare handoff
  - confirm publication

Зачем:

- меньше прокрутки на длинных задачах

### 15. Collapsible sections

Что добавить:

- сворачиваемые блоки:
  - source files
  - metrics
  - critic report
  - edit history

Зачем:

- уменьшить визуальную перегрузку

## Технические улучшения

### 16. Нормализованная модель edit history

Что можно сделать позже:

- вынести историю правок из `quality_report` в отдельную таблицу

Например:

- `content_item_revisions`
  - `id`
  - `content_item_id`
  - `previous_body`
  - `next_body`
  - `created_by`
  - `created_at`

Зачем:

- лучше для аналитики
- лучше для diff/revert/audit
- чище, чем хранить длинную историю в JSON

### 17. Серверный diff

Что можно сделать позже:

- вычислять diff на backend
- хранить его вместе с revision

Зачем:

- быстрее отрисовывать историю
- меньше клиентской логики

### 18. Optimistic UI для сохранения текста

Что добавить:

- локальное optimistic update
- rollback при ошибке

Зачем:

- editor будет ощущаться заметно быстрее

## Отдельный backlog для MCP

### 19. MCP tool для обновления текста publication task

Что добавить:

- tool вроде `ba_update_publication_task_content`

Зачем:

- чтобы Claude мог не только читать задачу, но и править publish-ready body напрямую

### 20. MCP tool для revert revision

Что добавить:

- tool для возврата к одной из прошлых версий текста

Зачем:

- удобно для совместной редакторской работы через Claude

## Рекомендуемый порядок реализации

Я бы делал так:

1. `Восстановить эту версию`
2. `Dirty state + unsaved warning`
3. `Autosave`
4. `Diff between revisions`
5. `Channel-specific preview`
6. `Normalized revision table`
7. `MCP update/revert tools`

## Короткий вывод

Сейчас publication workspace уже стал рабочим редакторским экраном. Следующая цель — превратить его из “формы подтверждения публикации” в полноценную редакторскую консоль:

- писать
- сравнивать
- откатывать
- проверять
- публиковать
- анализировать feedback

в одном месте.
